## 2026-04-18 - Semantic Tabs for Bottom Navigation
**Learning:** Interactive elements like bottom navigation tabs should use semantic `<button>` elements with `role='tab'` instead of `<div>` to ensure screen readers properly announce them as interactive tabs rather than generic text blocks. Sighted keyboard users also need visible focus indicators, so when resetting native button styles with `outline: none`, a `:focus-visible` fallback must be provided.
**Action:** When implementing custom navigation or tab components, ensure the container has `role="tablist"`, items use `<button>` tags with `role="tab"` and `aria-selected` attributes, non-semantic icons have `aria-hidden="true"`, and focus states are clearly visible via `:focus-visible`.
## 2026-05-13 - Focus Management and Semantics for Custom Overlay Menus
**Learning:** Custom overlay menus built with Vanilla JS and CSS `display: block/none` require explicit semantic roles (`role="dialog"`, `aria-modal="true"`) to communicate their purpose to screen readers. Furthermore, focus must be manually managed: when opened, focus should shift to the first interactive element (using `requestAnimationFrame` to ensure the DOM has updated), and users must be able to close the menu via the `Escape` key, which should then return focus to the trigger element to preserve keyboard navigation flow.
**Action:** When implementing custom overlay menus, always add `role="dialog"` and `aria-modal="true"`, manage focus shifting on open with `requestAnimationFrame`, and implement an `Escape` key handler that closes the menu and returns focus to the trigger.
