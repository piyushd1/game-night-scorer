## 2026-04-18 - Semantic Tabs for Bottom Navigation
**Learning:** Interactive elements like bottom navigation tabs should use semantic `<button>` elements with `role='tab'` instead of `<div>` to ensure screen readers properly announce them as interactive tabs rather than generic text blocks. Sighted keyboard users also need visible focus indicators, so when resetting native button styles with `outline: none`, a `:focus-visible` fallback must be provided.
**Action:** When implementing custom navigation or tab components, ensure the container has `role="tablist"`, items use `<button>` tags with `role="tab"` and `aria-selected` attributes, non-semantic icons have `aria-hidden="true"`, and focus states are clearly visible via `:focus-visible`.

## 2026-05-10 - Accessible Custom Overlay Menus
**Learning:** Custom overlay menus (like the host menu) need explicit semantic roles since they don't use native `<dialog>` elements. They must use `role="dialog"`, `aria-modal="true"`, and have an `aria-label`. Additionally, they must manage focus by shifting it to the first interactive element upon opening, and trap/restore focus properly when closing (e.g. via Escape key or backdrop click). Screen readers will otherwise not announce the context, and keyboard users can lose their place.
**Action:** When creating custom popups or overlays, add `role="dialog"`, `aria-modal="true"`, and an `aria-label`. Store `document.activeElement` before opening, use `requestAnimationFrame` to shift focus to the first item, provide `:focus-visible` styling for the items, add a document-level `Escape` key listener, and restore focus to the original trigger when closing.
