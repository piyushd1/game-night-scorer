## 2026-04-18 - Semantic Tabs for Bottom Navigation
**Learning:** Interactive elements like bottom navigation tabs should use semantic `<button>` elements with `role='tab'` instead of `<div>` to ensure screen readers properly announce them as interactive tabs rather than generic text blocks. Sighted keyboard users also need visible focus indicators, so when resetting native button styles with `outline: none`, a `:focus-visible` fallback must be provided.
**Action:** When implementing custom navigation or tab components, ensure the container has `role="tablist"`, items use `<button>` tags with `role="tab"` and `aria-selected` attributes, non-semantic icons have `aria-hidden="true"`, and focus states are clearly visible via `:focus-visible`.
## 2026-05-11 - Accessible Custom Overlay Menus
**Learning:** Custom overlay menus implemented in Vanilla JS must support accessibility by explicitly defining `role="dialog"` and `aria-modal="true"`. Furthermore, they must gracefully shift focus to the first interactive element inside the menu when opened (often requiring `requestAnimationFrame` to ensure the DOM structure is physically focusable when using `display: block`), and support closure via the `Escape` key, and restore focus to the trigger element when closed.
**Action:** When implementing custom overlay menus, add semantic `dialog` roles, explicitly capture `document.activeElement` prior to display, use `requestAnimationFrame` to shift focus inward upon opening, add a document-level `keydown` listener for the `Escape` key (guarded by state flags to avoid duplication), and restore focus to the captured trigger upon closing.
